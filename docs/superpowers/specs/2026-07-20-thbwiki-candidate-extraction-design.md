# THBWiki 角色候选数据抽取工具 —— 设计文档

- 日期：2026-07-20
- 状态：待实现（第一轮：纯抽取，不接 diff/写库）
- 关联：`docs/superpowers/specs/2026-06-08-candidate-management-design.md`（candidate 管理后台设计，定义了 `candidate_character` 的字段与导入契约）

## 1. 背景与问题

`candidate_character` 表（后端）和 `Touhou-Vote/packages/shared/data/character.ts` 的 `characterList`（前端）都是人工维护的"候选角色清单"，每年办投票前都要手动核对 THBWiki 上的角色信息，容易漏、容易错，且两边各维护一份、互不同步。

目标：做一个实验性、可反复运行的工具，从 THBWiki 抽取角色结构化信息，作为未来生成/核对候选清单的数据源。

**本轮范围明确收窄**：
- 只做角色（人物），不做作品/音乐目录（作品目录留到下一轮迭代）。
- 只做「抓取 → 抽取 → 落盘结构化数据」，**不**对接 `candidate_character` 表或 `character.ts` 做 diff、**不**自动写入任何正式数据源。diff/审核环节是明确的下一轮工作，本文档不展开设计。

## 2. 网络层预研结论（实测，2026-07-20）

在动手写抓取代码前先验证了 THBWiki 是否可爬、怎么爬最合适，结论如下：

- **域名**：面向用户的渲染页面在 `thbwiki.cc`，但真正的 MediaWiki 站点、`api.php` 在 `thwiki.cc`（`action=query&meta=siteinfo` 确认 `sitename: THBWiki`，MediaWiki 1.39.10）。`thbwiki.cc/api.php` 实测不可用。仓库里现有的 `src/apps/scraper/sites/thbwiki.py` 也是用 `thwiki.cc/api.php`，与实测结论一致。
- **robots.txt**（两个域名内容逐字节相同）：只 disallow 特殊命名空间（分类:/属性:/特殊:/MediaWiki:）和 diff/oldid 类查询串，本工具要访问的页面（角色列表数据页、角色词条页）都不在禁止范围。声明 `Crawl-delay: 60`。
- **走 API 而不是抓渲染 HTML**：`action=query&prop=revisions&rvprop=content&rvslots=main&titles=A|B|C` 实测可用，且**一次请求最多批量拿 50 个页面标题的原始 wikitext**（`paraminfo` 确认 `limit: 50`，已用 2 个标题实测成功）。这比"抓 HTML 再清洗成 Markdown 再解析表格"更可靠，也把请求数从"每页一次"降到"全量角色只需约 10~15 次请求"。因此**取消了原设计里的 HTML→Markdown 清洗层**，直接对 wikitext 做结构化解析（见第 4 节）。
- **SMW 语义查询（`action=ask`）对角色词条不适用**：实测"种族/职业/初登场"这几个语义属性在角色词条上是空的——SMW 属性是给作品词条（专辑/发售日期等）准备的，角色信息只是普通 wikitext 模板参数，不能走这条路。仓库里 `thbdata()` 用的正是这条路，但那是给作品条目用的，跟本工具要抽的角色字段是两码事，不能直接复用其查询逻辑。
- **限速策略**：3 次连续请求实测均 200、无 429/Retry-After 迹象，服务端似乎没有强制限流。但既然批量后总请求数已经降到 10~15 次，遵守 `Crawl-delay: 60` 的代价只是多花 10~15 分钟，没有理由不遵守——**本工具按 60 秒/请求的间隔执行**，不采用仓库现有生产代码里那条更激进的 0.1 秒间隔（那是给单条即时查询场景用的，使用场景和请求量级都不一样，不构成"可以更快"的先例）。
- **版权**：`siprop=rightsinfo` 显示 CC BY-NC-SA 3.0。本工具只提取结构化字段（人名、别名、分类、所属作品），不搬运正文/图片，非商业同人投票站场景下风险低；如果未来涉及搬运正文描述或图片，需要额外考虑署名义务，本轮不涉及。

## 3. 数据源结构（实测确认）

不是从"纯文字列表"渲染页解析——那个页面的角色数据是通过 `{{官方角色文字列表信息|角色集合={{#lsth:官方角色列表/角色数据|分节名}}}}` 动态transclude 出来的，原始 wikitext 里看不到具体角色名，只有模板调用。

真正的结构化数据源是它 transclude 的**源数据页** `官方角色列表/角色数据`，wikitext 是维护者手写的、按分节组织的类 CSV 格式：

```
<!--角色词条名,角色中文名,角色日文名,角色英文名,立绘1,立绘2,角色初登场,角色别名,角色分类,角色标签,角色介绍-->

== 主角二人组 ==
博丽灵梦,,,,{{立绘获取|...}},,,红白，赤色杀人魔,0,正作游戏,博丽神社巫女

== 游戏 ==
=== 东方灵异传 ===
博丽灵梦（旧作角色）,,,,{{立绘获取|...}},,,博丽巫女,1,旧作游戏,维护梦与传统的巫女
```

字段含义（按注释）：`角色词条名`（对应页面标题，必填，用于二次查询）、`角色中文名/日文名/英文名`（多数留空，表示"从词条页自动获取"）、`角色初登场`、`角色别名`（人工补充的俗称，可能非空）、`角色分类`（数字编码）、`角色标签`（如"正作游戏"/"旧作游戏"，粗分类）、`角色介绍`（一句话简介）。`==`/`===` 标题即角色所属的分节路径（如"游戏/东方灵异传"），对应候选字段里的 `work`。

单独的角色词条页（如"博丽灵梦"）的 wikitext 里有 `{{官方角色信息 | 人物名=... | 日文名=... | 日文读音=... | 英文名=... | 种族=... | 职业=... | 能力=... }}` 模板调用，是模板参数级别的精确键值对（已用"博丽灵梦"词条实测）。

**因此抽取分两跳，都走批量 wikitext API，不做链接发现**：
1. 抓 `官方角色列表/角色数据`（1 次请求）→ 解析出全部词条名列表 + 分节路径（work）+ 分类/标签 + 人工填写的别名。
2. 用第 1 步拿到的词条名列表，按 50 个一批批量抓每个词条页的 wikitext（约 10~15 次请求）→ 解析 `{{官方角色信息}}` 模板参数，补全中文名/日文名/日文读音/英文名/种族/职业/能力等字段。

## 4. 流水线结构

```
[抓取层 fetch] → [抽取层 extract] → [落盘 output]
```

### 4.1 抓取层

- `thwiki.cc/api.php`，`action=query&prop=revisions&rvprop=content&rvslots=main`，按 50 个标题一批。
- 请求间隔 60 秒（遵守 `Crawl-delay`），第 1 步（数据源页）和第 2 步（批量详情页）之间也算一次请求间隔。
- 幂等：维护 `fetch_manifest.json`（`title -> {revision_id 或 content_hash, fetched_at}`）。重跑时先抓数据源页判断内容是否变化：
  - 未变 → 复用上次的词条名/分节列表，跳过第 2 步里内容没变的词条（按 revision_id 比对，MediaWiki API 返回自带 `revid`，比自算 hash 更省事）。
  - 只有变化的词条才重新走抽取。
- 缓存目录（`fetch_manifest.json`、原始 wikitext）不进 git。

### 4.2 抽取层

**规则抽取（主力，无 LLM 依赖）**：
- 解析 `官方角色列表/角色数据` 的类 CSV 行 → 得到词条名、`work`（分节路径）、`角色标签`（映射到粗分类）、人工 `角色别名`（逗号分隔）。
- 解析每个词条页的 `{{官方角色信息}}` 模板参数 → 得到 `name`（人物名）、`origname`（日文名）、日文读音、`英文名`、种族/职业原文（`type_raw`）、能力原文。
- 消歧义/特殊页面（无 `{{官方角色信息}}` 模板的词条）：降级为仅用数据源页那一行的信息，标记 `partial_extraction: true`，不强行套模板解析。

**LLM 兜底（缩小范围，仅用于规则解析不出的部分）**：
- 将种族/职业原文（`type_raw`）归一化到受控词表 → `type`。受控词表本轮先用一份小的初始清单（人类/妖怪/神灵/幽灵/其他等大类，基于种族而非职业），实现时从已抓到的 `type_raw` 样本里人工归纳补全，不预先假设完备。
- 数据源页 `角色别名` 字段是逗号分隔的人工文本，规则上直接按顿号/逗号切分即可，通常不需要 LLM；仅当切分结果明显异常（如包含说明性文字而非纯别名）时才走 LLM 兜底。
- 每次 LLM 输出都必须附带来源原文片段，禁止无来源凭空生成字段值。
- LLM 调用失败/超时：该字段留空，标记来源为 `llm_failed`，不重试到阻塞整体流程。

`date`（首次登场年月日）本轮**不做**——数据源页的"角色初登场"字段多数为空（自动获取），而自动获取的来源（角色页面本身或所属作品的发售日期）本轮没有现成映射，强行做会拉出一个新的子项目，超出范围，先留空。

### 4.3 输出

每个角色一条记录，结构大致如下：

```json
{
  "name": "博丽灵梦",
  "origname": "博麗　霊夢",
  "altnames": ["Hakurei Reimu", "はくれい　れいむ", "红白", "赤色杀人魔"],
  "type": "human",
  "type_raw": "人类 / 巫女",
  "work": ["主角二人组"],
  "source_title": "博丽灵梦",
  "source_revid": 123456,
  "extracted_at": "2026-07-20T00:00:00Z",
  "extraction_method": {"work": "rule", "type": "llm_assisted", "altnames": "rule"},
  "partial_extraction": false
}
```

- 数据源页收录的角色全部抽出，不做"是不是候选人"的取舍——取舍是下一轮 diff 阶段的事。
- 单个词条抓取失败或没有预期模板：计入结果里的 `failed`/`partial` 列表（词条名 + 原因），不中断整体流程。
- 落盘两份：完整 JSON 数组 + 人类可读的 Markdown 汇总（按 `work` 分组的表格，便于肉眼扫抽取质量）。
- 输出目录：`docs/scraper/candidate-extraction/`，明确标注为实验性产出、非正式数据源，不直接消费于生产代码。

## 5. 代码落地位置

- `thvote-be-re/src/apps/scraper/candidate_sync/`：与现有 `src/apps/scraper/sites/thbwiki.py`（社媒统计爬虫）同域下的姊妹能力。内部按职责拆分：
  - `fetch.py`：批量 wikitext 抓取 + manifest 缓存 + 限速
  - `extract_wikitext.py`：类 CSV 数据源页解析 + `{{官方角色信息}}` 模板参数解析（规则抽取）
  - `extract_llm.py`：LLM 兜底抽取（type 归一化、异常别名兜底）
  - `output.py`：落盘 JSON/Markdown
- `thvote-be-re/scripts/`：一个薄的 CLI 入口脚本，串联以上模块，供手动/定期触发。

## 6. 明确不做的事（本轮范围外）

- 不对接 `candidate_character` 表或 `character.ts`，不做 diff，不做任何写入。
- 不做作品/音乐目录抽取。
- 不做 `date`（首次登场日期）抽取。
- 不抓渲染 HTML、不做 HTML→Markdown 清洗（API 直接给 wikitext，用不上）。
- 不做开放式链接发现/BFS 抓取。
- 不复用 `~/BFV/data-curation` 的任务队列/page_id 存储机制，也不复用其 HTML 清洗规则（原设计里计划复用，预研后发现用不上）。
- 不复用仓库现有 `thbdata()` 的 SMW `ask` 查询逻辑（该语义属性对角色词条不适用）。

## 7. 测试计划

- 规则抽取器：用固定的样本 wikitext（本文档已实测的"官方角色列表/角色数据"片段、"博丽灵梦"词条的 `{{官方角色信息}}` 片段）做单元测试，覆盖正常情况、无模板的降级情况、别名切分的边界情况。
- LLM 兜底：mock LLM 响应做单元测试，覆盖正常提取、格式异常、超时/失败三种情况，确认失败不阻塞整体流程。
- 端到端：跑一次真实小范围抓取（如只抓"主角二人组"分节的角色），人工核对输出 JSON/Markdown 的准确性，同时验证 60 秒限速与 manifest 幂等逻辑确实生效。
